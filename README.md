# CryptoPulse.AI — Personalized Crypto Investor Dashboard

A personalized crypto dashboard that gets to know the user through a short onboarding quiz, then serves daily AI-curated content tailored to their interests — with thumbs up/down feedback on every section, stored for future model improvements.

**Live app:** _<deployed URL here>_
**Repository:** https://github.com/alinplotnik/crypto-advisor
**Database access:** Supabase (PostgreSQL) — credentials shared separately on request.

## Features

- **Auth** — email/password signup and login (Supabase Auth, cookie-based sessions). A signup trigger auto-creates the user's profile row.
- **Onboarding quiz** — assets of interest, investor type, preferred content. Saved as preferences and used to personalize every dashboard section.
- **Daily dashboard**, four sections, each with 👍/👎 voting stored in the DB:
  - 💰 **Coin Prices** — live prices + 24h change for the user's assets (CoinGecko)
  - 📰 **Market News** — live RSS feed, with the user's assets prioritized to the top
  - 🤖 **AI Insight of the Day** — generated by Gemini, personalized and grounded in today's real market data
  - 😂 **Fun Crypto Meme** — a fresh meme from r/cryptocurrencymemes on every page load

## Interesting decisions along the way

**Every external section has a fallback, and every failure is observable.** Each dashboard section is a server component streamed inside its own `<Suspense>` boundary, so one slow/failing API never blocks the page. News falls back to a curated static list personalized by the user's assets; memes fall back to images bundled in `/public`; prices and insights degrade to a friendly message. All failure paths log the actual status/body so problems are diagnosable in production logs instead of failing silently.

**I dropped OpenRouter for Gemini after measuring, not guessing.** The free OpenRouter model I first picked returned HTTP 200 in ~10s but took 60+ seconds to finish generating — my 20s timeout never had a chance. Gemini 2.5 Flash answers the same prompt in ~5s. The integration retries on 503 (model overloaded) with a short backoff, fails fast with a clear log when the API key is missing, and times out at 20s.

**The AI insight is grounded in real data, not hallucinated "news".** The model has no web access, so asking it "what happened today" would invent facts. Instead, the prompt is built from today's date plus a live CoinGecko snapshot of the user's own assets (price + 24h move), and the model must cite at least one concrete number, tailored to how the user's investor type thinks. Different data every day → genuinely different, fact-based insight every day.

**Insights are cached per user per day in the DB** (`insights` table, unique on `user_id + insight_date`) so a user can refresh the dashboard all day while consuming exactly one LLM generation. I considered deduplicating generations across users with identical profiles, but with 3 investor types × 2⁶ asset combinations the real-world cache-hit rate would be near zero — and per-user generation is the architecture that feedback-driven personalization needs anyway (see bonus section below). The write uses an upsert so two concurrent page loads can't race into a constraint violation.

**Reddit scraping is dead; I verified it and chose accordingly.** Reddit returns 403 to non-OAuth requests even from residential IPs (worse from cloud IPs). Rather than ship scraping that breaks on deploy, the meme section uses meme-api.com (no auth) with NSFW filtering, fetched with `cache: 'no-store'` so the meme changes on every dashboard refresh, plus bundled fallback images so the section can never break.

**Row Level Security is enabled and *proven*, not just enabled.** Policies restrict every table to `auth.uid() = user_id` and mirror the app's exact query patterns (e.g. `insights` deliberately has no UPDATE policy — its upsert uses `ON CONFLICT DO NOTHING`). I verified the policies with an automated two-user test: user A cannot read, forge, or update any of user B's rows across all four tables (9/9 checks), and anonymous requests see nothing. The signup trigger is `security definer`, so profile creation still works under RLS.

**Votes are upserts, not inserts.** `feedback` is unique on `(user_id, section, content_id)` — re-voting updates your existing vote instead of stacking duplicates, which keeps the table clean as training data (one current opinion per user per item).

|---|---|

## Bonus — how feedback enables future model improvements

The feedback loop was designed so today's thumbs become tomorrow's training data:

**What's already stored.** Every vote is `(user_id, section, content_id, vote, created_at)`. Crucially, `content_id` points to the *exact* content: the insight's date (whose full generated text is preserved in `insights`), the news article URL, the meme's Reddit post. Joined with `preferences`, each vote becomes a labeled example: *"a HODLer interested in Bitcoin+ETH disliked this cautionary insight."* Without storing the generated text itself, a 👎 would be meaningless later — that's why the `insights.content` column exists.

**Stage 1 — prompt-level improvement (cheap, immediate).** Aggregate a user's recent insight votes and inject style guidance into the daily prompt ("this user downvoted macro commentary, upvoted on-chain metrics"). No training at all — just retrieval into the prompt — and it can ship within a day.

**Stage 2 — learned re-ranking for news and memes (classic ML).** Both sections choose from candidate pools, which is a ranking problem: embed candidates (title/content) and the user profile, train a lightweight ranker (logistic regression or a small two-tower model) on the accumulated vote labels, and re-rank each pool per user. As vote volume grows, collaborative filtering ("users with similar vote patterns also liked…") layers on top of the cold-start content-based profile.

**Stage 3 — preference-tuning the insight model (when volume justifies it).** Per-user daily insights with votes are exactly the shape preference optimization (DPO/RLHF-style) needs: prompt + chosen/rejected completions. Pairs are formed from upvoted vs. downvoted insights generated for similar profiles, and a small open model is fine-tuned to replace or complement the API model.

**What I'd add to the schema next** (deliberately not implemented, per the task):
- An **impressions table** — content shown but *not* voted on. Most users don't vote, and "saw it, skipped it" is a weak negative signal that dwarfs explicit votes in volume.
- **Prompt/model version columns** on `insights` — so feedback is attributable to the prompt that produced it, enabling A/B tests between prompt variants ("votes per impression, v1 vs v2") with the votes already being collected.

## AI tools used

This project was built in collaboration with **Claude Code** (Anthropic's CLI agent). A summary of how it was used:

- **Scaffolding and implementation** — dashboard sections, onboarding flow, auth pages, API routes, migrations, and the theme were built iteratively through conversation; I directed the architecture, reviewed every change, and made the product decisions (e.g. which questions to ask, fallback behavior, what to commit vs. keep local).
- **Evidence-based debugging** — when the AI insight kept failing, the agent measured the OpenRouter free model directly (200 OK in 10s, body still streaming at 60s) before we switched to Gemini, which it verified at ~5s. The Reddit-403 discovery that shaped the meme section came from the same test-first approach.
- **Verification beyond "it compiles"** — it wrote local scripts (kept out of the repo) that proved the Gemini integration end-to-end, validated RLS with two throwaway users across 9 cross-access checks, and drove the real app with Playwright (signup → onboarding → dashboard) taking screenshots to verify the UI overhaul.
- **Honest trade-off discussions** — per-user vs. per-profile insight caching, scraping vs. APIs for memes, and the RLS policy design were each decided after weighing alternatives, several of which are documented above.
